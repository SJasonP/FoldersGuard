# FoldersGuard Architecture

## Design Principle

FG uses a hierarchical key model. Files are encrypted independently. FG's own encrypted database stores metadata that
describes folders and files and contains the internal keys needed to open them.

The folder is the central authorization boundary. Users do not normally handle raw keys directly; they unlock data with
passwords or FG share database files. Unlocking a folder grants access to that folder and everything beneath it, but not
to unrelated folders or files.

FG creates one active project from one top-level folder at a time. Share databases may describe a rootless shared
payload instead of a normal project root.

Encrypted content and FG data are separate. The encrypted content tree is the portable storage payload. FG data is the
private encrypted project database that maps UUID names to real names, stores metadata records, tracks internal keys,
and records the relationship between original structure and encrypted structure.

The FG app id is `com.SJasonP.FoldersGuard`. The FG data directory name is `FoldersGuard`. Each FG project also has its
own generated project id.

## Core Concepts

### File

A file is the smallest independently shareable content object.

- Each file has one internal file key.
- The file key encrypts the file content.
- A file may be physically stored as one encrypted object or as multiple balanced parts.
- Parts are not independently encrypted or authorized.

### Folder

A folder is the smallest recursively shareable collection.

- Each folder has one internal folder key.
- Folder keys define logical authorization boundaries and are included in share-scoped data when that folder is shared.
- Folder records identify child relationships inside the unlocked FG database.
- Sharing a folder includes the metadata and internal keys needed to restore that folder recursively.
- The top-level project database must have a password.

### Top-Level Folder

The top-level folder is mandatory and unique for each active project.

Rules:

- `fg encrypt` creates one active project from one top-level folder.
- Project metadata operations use that project root as the normal operating boundary.
- `fg decrypt` and `fg verify` can operate on either one active project or one `.fgs` shared payload.
- The top-level project database password is required.
- Unlocking the top-level folder allows recursive access to all supported contents under it.

### FG Data

FG data is separate from encrypted content.

V1 stores active FG data only in FG's data directory. Users may export FG data for backup or transfer workflows.

FG data is stored as a SQLCipher-encrypted SQLite database. The database encryption is the primary protection for names,
metadata, search indexes, and internal keys at rest.

FG data contains enough information to:

- Unlock the top-level folder with its password.
- Restore cleartext folder and file names.
- Map real names to visible UUID names.
- Track child type: file or folder.
- Restore captured filesystem metadata for supported files and directories.
- Store file keys for direct child files.
- Store folder keys for direct child folders.
- Store file sizes and part layout.
- Verify integrity.
- Rename entries without requiring encrypted content to be present.
- Generate manual storage operation instructions for upload, move, and delete workflows.

### Password

Passwords are the only user-facing unlock mechanism.

Internally, FG generates random high-entropy keys for files and folders. Passwords unlock encrypted FG databases through
SQLCipher's password-based keying.

The top-level project database must have a password. Share databases may have a password or be opened without a
password.

### Share Database

A share database is an FG proprietary SQLCipher-encrypted database generated by the sender.

It accompanies encrypted content when sharing selected files or folders. It contains the FG data subset needed to
restore the shared content, including access to the relevant internal keys. A share database may be password-protected
or unprotected.

From the user's point of view, a share database has no single required root. It may describe one file, multiple files,
one directory, multiple directories, or a mixed top-level set. The v1 implementation may use an internal virtual root
only as a schema device; that virtual root is not visible when restoring shared content.

If a share database is unprotected, it must be treated as a bearer secret: anyone who has both the encrypted content and
the share database can decrypt the shared content.

Share generation always produces `.fgs`.

### Database Extensions

`.fg` is reserved for a normal FG project database with exactly one top-level object, and that object must be a
directory.

`.fgs` is used for share databases and every database shape outside the normal `.fg` case.

### Visible UUID Name

All encrypted files and directories use generated UUID names in native mode. The mapping from UUID to real name is
stored in encrypted FG databases or share databases.

This hides names while keeping the folder hierarchy visible and usable through ordinary file managers and cloud drives.

## Content Tree Invariance

The encrypted content tree preserves the original logical structure:

- A cleartext directory maps to an encrypted directory.
- A cleartext file maps to an encrypted file, unless it must be split.
- With split parts treated as one logical file, the encrypted tree has the same logical folder and file structure as the
  original tree.

When a file is split, the split representation is still one logical file in FG data. The physical representation may
require multiple storage objects, but FG must not expose those parts as independent user-visible files in its logical
model.

## Key Hierarchy

```text
Project password
  unlocks SQLCipher project database
    database contains root metadata
    database contains file keys for root files
    database contains folder keys for child folders
      child folder key marks a recursive authorization boundary
        share generation can include the child's subtree metadata and keys
```

## Sharing Model

### Sharing A Single File

To share one file, the sender provides:

- The encrypted file object or its parts.
- An FG share database for that file.

The share database must include enough metadata and internal key material for FG to reassemble and decrypt the file. It
must not reveal access to sibling files or folders.

### Sharing A Folder

To share one folder, the sender provides:

- The encrypted folder tree starting at that folder.
- An FG share database for that folder.

The recipient uses FG and the share database to decrypt the shared folder metadata, then recursively discover and
decrypt all children.

This must not reveal access to parent folders, sibling folders, or unrelated files.

### Sharing Selected Items

To share multiple selected items, the sender provides:

- The encrypted content objects or encrypted folder trees for those selected items.
- One FG share database for the selected set.

The share database may describe one file, multiple files, one folder, multiple folders, or a mixed set. From the
recipient's point of view, the shared payload has no required single visible root.

The share database must include enough metadata and internal key material for FG to restore only the selected items and
their descendants.

Selecting both a folder and one of its descendants does not duplicate the descendant in the share payload.

This must not reveal access to unselected parent folders, sibling folders, sibling files, or unrelated content.

## Native Encryption Requirements

Native mode uses the v1 encryption suite. Decryption fails if ciphertext, nonce, authentication tag, FG database
content, or part order is tampered with.

V1 encryption suite:

- AES-256-GCM for file content and part content encryption.
- SQLCipher for FG project and share databases.
- Random keys generated with a cryptographic random source.
- Per-object nonces that are never reused with the same key.
- 256-bit random file keys and folder keys.
- Explicit versioning and algorithm identifiers in database metadata and content headers.

The security model depends on strong user passwords, SQLCipher password-based database keying, 256-bit symmetric keys,
and authenticated encryption.

## FG Reserved Data

FG reserved files and data do not need name hiding. Revealing that data is protected by FG is acceptable because
confidentiality must depend on passwords, share databases, and authenticated encryption rather than obscurity.

Reserved data is authenticated and versioned. Attackers may know where FG data and share databases are located, but they
must not be able to decrypt or modify them undetected.

## Metadata-Only Operations

Because FG data is separate from encrypted content, some operations do not require the encrypted content to be present.

### Rename

Renaming a protected file or folder changes only FG data. The encrypted content UUID path remains unchanged.

This supports workflows where encrypted content is stored remotely and unavailable locally.

### Add, Move, Delete

Adding, moving, or deleting content usually requires physical changes to encrypted content storage. FG supports
manual-storage workflows by generating clear instructions.

When FG has direct filesystem access to the encrypted content, it may perform these operations itself. The design must
still support the manual instruction workflow.

## Planned Operations And Data

This section records the architectural decisions for the planned feature set. It is planned and not yet implemented. The
behavior-level requirements are in the product requirements planned feature set.

### Operation Resumption

A long-running content operation can be re-run after an interruption and continue rather than restart.

- Encrypted objects live at deterministic visible UUID paths derived from the project plan, so whether an object is
  already written is answerable from the plan and the filesystem alone, without extra resume state on disk.
- On resume, encryption treats an object as complete when its visible path exists and passes integrity verification. A
  present but corrupt or partially written object is rewritten.
- On resume, decryption and restore treat an output as complete when the output path exists and matches the expected
  decrypted content.
- Source-file deletion under delete-source remains safe: a source file is deleted only after its encrypted object is
  confirmed complete in the same run, or verified complete on resume.
- Resumption verifies an existing object before skipping it. A faster skip-by-presence option may be offered; the
  integrity-verifying mode is the default.
- If the plan changed between runs, such as a different split size or added or removed items, resumption applies only to
  objects that still match the current plan. Objects that no longer match are rewritten or removed to match the plan.

### Operation Failure Policy

Content operations choose between aborting on the first error and continuing past recoverable item-level failures.

- The default policy is abort on the first error, matching v1 behavior.
- Continue-on-error policy records each item-level failure and proceeds with the remaining items.
- Item-level failures are recoverable conditions tied to one file, such as an unreadable source or a permission error on
  one entry.
- Fatal conditions abort regardless of policy: a full output disk, a database error, or any condition that threatens the
  integrity of the project database or the whole operation.
- A source file is never deleted when its own encryption failed.
- The operation result reports the number of failed items and a per-item reason. Internal keys and passwords are never
  included in failure detail.

### Change Password

Changing a project or share password re-keys the database without re-encrypting content.

- Content is encrypted with internal per-file and per-folder keys, and the password protects only the database, so
  changing the password leaves every encrypted object unchanged.
- A password change does not require encrypted content to be present.
- The operation is crash-safe and never leaves the user without a working database. The sequence is: back up the
  database, re-key a copy, confirm the copy opens under the new password, then atomically replace the live database.
- The old password is verified before the change proceeds, and the new password is confirmed.
- A share database password change protects only future copies. Share databases already distributed are independent
  copies and are unaffected.

### Metadata-Database Backup

FG snapshots a project database before operations that modify or replace it.

- A backup is taken before applying project changes, deleting a project, and changing a password.
- Backups live in a managed backups location inside FG's data directory and are pruned to a configurable retention
  limit.
- A backup is a copy of the encrypted SQLCipher database; it contains key material and is stored with the same file
  permission restrictions as the live database.
- FG can restore a project database from a backup.

### Encryption Concurrency

File encryption may process multiple files concurrently.

- Each file is encrypted independently with its own random key and per-object nonces, so concurrent file encryption
  shares no cipher state and does not affect the encryption suite's guarantees.
- Concurrency is bounded by a worker pool whose size defaults to a value derived from the host CPU count and is
  configurable.
- Concurrency is across files; chunk streaming within a single file remains sequential.
- Folder-creation ordering and source-file deletion remain correct under concurrency: a folder exists before its
  children are written, and a source file is deleted only after its own encrypted object is complete.
- Progress remains byte-weighted, accurate, and monotonic under concurrency.

## Directory Hierarchy Policy

V1 preserves hierarchy. This is intentional.

Benefits:

- Users can manually upload a protected folder to a cloud drive.
- Users can manually download a protected subfolder without resolving a global object database.
- Users can share one encrypted folder as a normal directory.
- The product behavior matches the name FoldersGuard.

Tradeoff:

- Observers can see the shape of the encrypted tree.
- Observers can count visible files and folders.
- Observers can estimate storage size.

## Filesystem Entry Policy

FG supports only regular files and directories.

Unsupported entries are ignored as if they do not exist:

- Symlinks.
- Sockets.
- FIFOs.
- Device files.
- Other special filesystem entries.

Unsupported entries are not represented in FG metadata and are not reported in normal command output.

Hard link relationships are not preserved. Each hard link path is processed as a normal regular file.

FG also recognizes platform-generated noise files. Noise files are regular files or directories that operating systems
and file managers commonly add around user content, but that users usually do not intend to protect as project content.

Recognized noise names:

- `.DS_Store`
- `._.DS_Store`
- Any AppleDouble sidecar beginning with `._`
- `Thumbs.db`
- `ehthumbs.db`
- `desktop.ini`
- `.Spotlight-V100`
- `.Trashes`
- `.fseventsd`

Noise handling is configurable:

- Ignore everywhere: default. FG treats recognized noise files as if they do not exist during source scanning, project
  creation, project add, encrypted content matching, verification, decryption, share restore, source cleanup, and
  output-folder emptiness checks. Ignored noise files are not represented in FG metadata.
  For directory cleanup, overwrite preparation, and empty-folder checks, ignored noise files do not count as user content
  and may be removed as incidental cleanup when FG removes or replaces the containing directory.
- Ignore during verification and matching: FG may record noise files from source content as normal regular files, but
  ignores extra recognized noise files that appear around encrypted content while matching, decrypting, restoring, or
  verifying.
- Do not ignore: FG treats recognized noise files as ordinary filesystem entries when they are regular files or
  directories. They may be stored in FG metadata, encrypted, restored, and reported as extra content.

## Filesystem Metadata Policy

FG preserves first-tier filesystem metadata for supported regular files and directories.

Metadata captured for restore:

- Item type.
- Real name.
- Parent-child structure.
- File size.
- Modification time.
- Access time when available.
- Creation time when available.
- Permission mode.
- Basic Windows file attributes when available.

Implementation model:

- FG records which metadata fields were captured for each item.
- Restore applies only recorded fields and tolerates host filesystem timestamp rounding.

Restore order:

- Create directories.
- Restore file contents.
- Apply file permissions and attributes.
- Apply file timestamps.
- Apply directory permissions, attributes, and timestamps after child entries are restored.

## Implementation Language

FG core is implemented in Go.

Reasons:

- Single binary distribution.
- Good cross-platform filesystem support.
- Mature standard library for cryptography, IO, path handling, and concurrency.
- Strong fit for CLI and local desktop WebUI integration.
- Smaller runtime and dependency surface than Node or Python for this kind of software.

Python may be used for auxiliary test tooling, but not for the core encryption implementation.

## User Interface Architecture

FG has two user-facing interfaces:

- CLI for stable automation.
- Local desktop WebUI for normal interactive use.

Both interfaces call shared Go application services. Encryption, decryption, database access, filesystem validation, and
storage operations must stay in the Go core.

The WebUI frontend must not implement cryptography, parse FG databases directly, or manipulate encrypted content
directly. It collects user choices, displays project state, and shows operation progress reported by the Go core.

Progress for long-running operations is owned and measured by the Go core. The core establishes byte totals before the
main work, reports byte-weighted progress through phases as content is streamed, and reports a terminal state of
completed or failed. The frontend renders this reported progress and must not infer progress by inspecting the
filesystem. Long-running operations cannot be cancelled; while one runs, the WebUI blocks closing the window and quitting
the app until it finishes.

The WebUI runs as a local desktop application and must not expose a general remote HTTP API by default.
